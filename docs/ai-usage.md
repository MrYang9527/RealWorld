## 二、主要功能的 AI 使用详情

### 2.1 后端 API 搭建

**核心 Prompt:**
```
基于 RealWorld 规范，使用 Express.js + SQLite 实现一个后端 RESTful API，
需要包含：用户注册/登录（JWT认证）、文章CRUD、评论管理、
文章收藏/取消收藏、用户关注、标签管理。
请使用 express-validator 做输入验证，better-sqlite3 做数据库操作。
```

**AI 生成内容：**
- Express 应用骨架
- 数据库表结构设计（7张表）
- 所有 API 路由的初步实现
- JWT 中间件
- 输入验证规则

**我的修改：**
1. 所有错误提示从英文改为中文（用户体验考虑）
2. 评论删除增加权限校验（AI 遗漏了"只能删自己的评论"）
3. slug 生成函数增加中文标题的兜底处理
4. 登录/注册返回字段统一，去除敏感信息
5. 文章列表增加 tag/author/favorited 组合筛选

### 2.2 前端框架搭建

**核心 Prompt:**
```
使用 React 19 + Vite + React Router v7 + Axios 搭建 RealWorld 前端。
```

**AI 生成内容：**
- 完整的组件树和路由配置
- API 客户端封装（Axios拦截器）
- AuthContext 认证状态管理
- 所有页面的初始代码

**我的修改：**

2. 配色方案从蓝色改为绿色（RealWorld 官方配色）
3. 收藏按钮增加乐观更新和错误回滚
4. 标签输入交互改为回车添加（AI 给的是逗号分隔）
5. 分页组件修复页码计算边界bug
6. 路由守卫增加 loading 状态避免首次加载闪烁

### 2.3 文章编辑器

**核心 Prompt:**
```
实现一个文章编辑器页面，支持标题、摘要、正文（textarea）、标签输入。
编辑模式下需要从API加载已有文章数据并回填表单。
```

**Prompt 演化过程：**
1. **初始 Prompt:** "写一个文章发布页面"
   - 问题：太笼统，AI 生成的表单缺少摘要字段，标签处理也不完整
2. **修改后 Prompt:** "文章编辑器，需包含 title/description/body/tagList，tagList用tag-input方式"
   - 改进：AI 补全了字段，但仍用逗号分隔标签
3. **最终 Prompt:** "标签用回车添加，已添加的标签显示为可删除的chip，编辑模式下回填数据"
   - 结果：符合预期的交互方式

**我的修改：**
- 标签输入组件完全手写（回车添加+点击删除）
- 编辑模式下的数据预填逻辑调整
- 提交后跳转到文章详情页而非列表页

## 三、AI 失败案例分析（至少3个）

### 失败案例 1：Express 错误处理中间件不生效

**场景：** AI 生成的后端入口使用了标准的 Express 4.x 错误处理中间件：
```javascript
app.use((err, req, res, next) => {
  res.status(500).json({ error: '服务器错误' });
});
```

**问题：** 当前环境安装的是 Express 5.x，Express 5 中同步抛出的错误不会自动传递给错误处理中间件，导致部分异常返回 HTML 而非 JSON。

**解决过程：**
1. 我先让 AI 检查为什么错误中间件不工作
2. AI 建议降级到 Express 4.x，但我阅读文档后发现 5.x 的改动可以接受
3. 最终手动在各路由中加了 try-catch 确保异常被正确捕获

**教训：** AI 不一定了解最新的框架版本变化，需要自己查看文档验证。

### 失败案例 2：React Router v7 路由不匹配

**场景：** AI 生成的 App.jsx 使用了 React Router 的配置方式：
```jsx
<Route path="/editor/:slug" component={ArticleEditor} />
```

**问题：** React Router v7 中，`component` prop 已被弃用，应使用 `element` prop。直接使用 `component` 会导致页面无法渲染且控制台无报错（静默失败）。

**解决过程：**
1. 花了半小时排查为什么编辑器页面空白
2. AI 反复检查组件代码但找不出问题
3. 最终我打开 React Router 官方文档，发现是 API 版本差异
4. 改为 `<Route path="/editor/:slug" element={<ArticleEditor />} />`

**教训：** 框架版本差异是 AI 的盲区，遇到路由问题要第一时间查官方文档。

### 失败案例 3：CSS 统一样式污染

**场景：** AI 生成的 CSS 直接对全局元素设置样式：
```css
input { width: 100%; padding: 12px; border-radius: 4px; }
button { background: #5cb85c; color: white; ... }
```

**问题：** 这些全局样式污染了导航栏、标签组件、分页等所有包含 input/button 的地方，导致导航栏的按钮也变成了绿色全宽。

**解决过程：**
1. AI 第一次修复只是加了 `.form-group` 前缀，但没覆盖所有input场景
2. AI 第二次又建议用 CSS Modules，但会导致需要改所有组件的引用方式
3. 我最终决定把所有 input 样式限定在 `.form-group input`、`.tag-input-wrapper input` 等具体选择器下，手动重写了整个 CSS

**教训：** AI 倾向于用粗粒度的全局样式快速出效果，但实际项目需要更细致的样式隔离。这是我手动修改量最大的部分。

## 四、Prompt 使用技巧总结

### 4.1 有效 Prompt 的特征

1. **明确技术栈和版本号** — 例如 "Express 5.x + better-sqlite3" 而非 "Node.js后端"
2. **描述输入输出格式** — "返回 `{ article: { slug, title, ... } }` 格式"
3. **给出约束条件** — "错误信息用中文"、"标签输入用回车而非逗号"
4. **分步骤提问** — 先让 AI 给出整体架构，再逐个模块细化

### 4.2 Prompt 演化实例

以"文章列表筛选"功能为例：

| 轮次 | Prompt | AI 输出质量 |
|-----|--------|-----------|
| 第1轮 | "实现文章列表API" | 只返回全部文章，无筛选 |
| 第2轮 | "文章列表支持按tag筛选" | 加了tag参数，但不支持组合筛选 |
| 第3轮 | "支持 tag/author/favorited 组合筛选 + 分页" | 实现了所有筛选，但SQL拼接有注入风险 |
| 第4轮 | "用参数化查询，limit和offset默认值20/0" | ✅ 最终满意版本 |

### 4.3 我的 AI 协作心得

1. **AI 适合生成框架代码，不适合处理细节逻辑** — 我让AI搭好骨架，然后手动调整业务逻辑和边界条件
2. **先让 AI 出方案，再让它写代码** — 直接要代码容易偏，先让它解释设计思路，确认后再要代码
3. **一个 Prompt 解决一个问题** — 一次提太多需求，AI 容易遗漏
4. **代码review不能省** — AI 生成的代码我每行都看过，很多细节需要改
